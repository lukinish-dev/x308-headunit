# AGENTS.md

## Назначение

Этот файл задаёт обязательные правила работы Codex в проекте `x308-headunit`.

Главные цели:

- минимальный расход токенов;
- минимальный объём чтения файлов и вывода команд;
- точечные изменения без лишнего рефакторинга;
- работа непосредственно в целевом окружении Rock Pi;
- предсказуемые и проверяемые изменения;
- сохранение стабильности проекта.

---

## 1. Общий принцип работы

Перед изменениями Codex должен:

1. Кратко определить, какая часть проекта относится к задаче.
2. Найти минимальный набор нужных файлов.
3. Прочитать только эти файлы и их непосредственные зависимости.
4. Внести минимальный совместимый патч.
5. Запустить только необходимые проверки.
6. Показать краткий итог и релевантный diff.

Запрещено начинать работу с полного обхода репозитория, массового чтения документации или архитектурного аудита, если этого прямо не требует задача.

---

## 2. Работа по SSH

Проект расположен на Rock Pi и считается основным рабочим экземпляром.

Правила:

- Работать непосредственно в репозитории на Rock Pi.
- Работа через SSH считается обычной локальной работой в удалённой файловой системе.
- Не копировать весь проект на другую машину только ради выполнения задачи.
- Не создавать дополнительную локальную копию проекта без явной необходимости.
- Сборку, тесты, Git-команды и запуск приложения выполнять там, где находится рабочий репозиторий и целевое окружение.
- Не синхронизировать весь проект между компьютером и Rock Pi, если можно работать напрямую по SSH.
- Не включать в итоговый отчёт полный журнал SSH-сессии.
- Показывать только релевантные команды, результаты и ошибки.

Сам факт работы по SSH не является причиной читать больше файлов, запускать больше команд или собирать дополнительный контекст.

---

## 3. Экономия токенов

Расход токенов определяется прежде всего объёмом:

- прочитанного кода;
- прочитанной документации;
- полученных логов;
- stdout и stderr команд;
- diff;
- повторно загруженного контекста.

Расположение проекта — локально или по SSH — само по себе не должно увеличивать объём контекста.

Обязательные правила:

- Читать только файлы, относящиеся к задаче.
- Не перечитывать уже изученные файлы без новой причины.
- Не повторять одинаковые команды без изменения входных условий.
- Не проводить повторный полный аудит после каждого изменения.
- Не загружать весь репозиторий в контекст.
- Не читать lock-файлы, сгенерированные файлы, сборочные артефакты и зависимости без необходимости.
- Не выводить полный stdout, если достаточно краткого результата.
- При успешной проверке достаточно статуса `PASS` и короткого резюме.
- При ошибке сначала читать последние релевантные строки.
- Расширять диапазон логов только если причина ошибки не найдена.
- Останавливать исследование, как только контекста достаточно для безопасного изменения.

## 4. Поиск по проекту

Предпочтительные инструменты:

```bash
rg "строка или символ" src/
rg --files src/
git grep "строка или символ"
git diff -- path/to/file
git log --oneline -- path/to/file
```

Для поиска файлов:

```bash
find path/to/specific/dir -maxdepth 2 -type f
```

Без необходимости запрещены команды с большим рекурсивным выводом:

```bash
find .
tree
ls -R
cat большого_файла
```

Если структура проекта неизвестна, сначала использовать ограниченный вывод:

```bash
find . -maxdepth 2 -type f | head -100
```

или:

```bash
rg --files | head -100
```

Не выводить содержимое `node_modules`, `.git`, `dist`, `build`, кэшей и временных каталогов.

---

## 5. Чтение файлов

Перед чтением большого файла сначала определить нужный участок через:

```bash
rg -n "имя функции|компонента|переменной" path/to/file
```

Затем читать только нужный диапазон строк.

Не читать файл целиком, если задача касается одной функции или одного компонента.

При необходимости расширять контекст постепенно:

1. целевая функция;
2. соседний код;
3. прямые импорты;
4. только затем более широкий контекст.

Не читать все импорты рекурсивно, если задача этого не требует.

---

## 6. Изменения кода

Codex должен вносить минимальный патч, достаточный для выполнения задачи.

Правила:

- Не менять архитектуру без явного запроса.
- Не переименовывать несвязанные сущности.
- Не форматировать целые файлы из-за изменения нескольких строк.
- Не менять стиль проекта без необходимости.
- Не обновлять зависимости без явной причины.
- Не выполнять массовый рефакторинг попутно.
- Не исправлять несвязанные проблемы, если они не блокируют задачу.
- Не создавать новые абстракции, если простое локальное изменение достаточно.
- Сохранять обратную совместимость, если пользователь явно не попросил иначе.
- Соблюдать существующие соглашения проекта.

Если обнаружена отдельная проблема, не относящаяся к задаче, упомянуть её кратко в итоговом отчёте, но не исправлять автоматически.

## 7. Команды и терминальный вывод

Избегать команд, создающих большой объём вывода.

Предпочитать:

```bash
command 2>&1 | tail -100
command 2>&1 | grep -E "error|warning|failed"
```

Для тестов запускать сначала минимальный релевантный набор:

```bash
npm test -- ComponentName
npm test -- path/to/test
pytest path/to/test.py
```

Полный набор тестов запускать только если:

- изменение затрагивает общую инфраструктуру;
- локальных тестов недостаточно;
- это прямо требуется правилами проекта;
- есть высокий риск регрессии.

Для сборки при ошибке сначала анализировать последние строки:

```bash
npm run build 2>&1 | tail -120
```

Не передавать модели тысячи строк успешных логов.

---

## 8. Проверки после изменений

Проверки должны соответствовать масштабу изменений.

Приоритет:

1. синтаксис и типы изменённых файлов;
2. релевантный тест;
3. локальная сборка затронутой части;
4. полный build или test suite только при необходимости.

Не запускать несколько эквивалентных проверок без причины.

Если проверка не запущена, явно указать почему.

Не утверждать, что изменение проверено, если команда фактически не выполнялась.

---

## 9. Git

Репозиторий уже связан с GitHub через SSH.

Основной remote:

```text
git@github.com:lukinish-dev/x308-headunit.git
```

Codex может:

- смотреть `git status`;
- смотреть точечный `git diff`;
- создавать изменения;
- подготавливать commit;
- создавать commit только по прямому запросу пользователя.

Codex не должен:

- выполнять `git push` без прямого запроса;
- делать force push;
- переписывать историю;
- удалять ветки;
- делать reset с потерей данных;
- включать в commit несвязанные изменения;
- выполнять `git add -A` вслепую.

Перед commit обязательно:

```bash
git status --short
git diff --stat
git diff -- path/to/changed/files
```

Добавлять в commit только файлы, относящиеся к задаче.

Предпочтительно:

```bash
git add path/to/file1 path/to/file2
```

вместо:

```bash
git add -A
```

Сообщение commit должно быть кратким и описывать фактическое изменение.

## 10. Безопасность рабочей копии

До изменений проверить:

```bash
git status --short
```

Если в рабочей копии есть чужие или пользовательские изменения:

- не перезаписывать их;
- не откатывать их;
- не включать их в свой patch;
- не использовать команды, способные потерять данные;
- работать только с целевыми файлами.

Запрещены без прямого запроса:

```bash
git reset --hard
git clean -fd
git checkout -- .
git restore .
git rebase
git push --force
```

---

## 11. Документация

Не обновлять документацию автоматически после каждого небольшого изменения.

Обновлять документацию только если:

- изменилось публичное поведение;
- изменилась установка или запуск;
- изменились переменные окружения;
- изменился внешний API;
- пользователь прямо попросил обновить документацию.

Не создавать длинные отчёты и дополнительные markdown-файлы без необходимости.

---

## 12. Зависимости

Перед добавлением новой зависимости проверить, можно ли решить задачу средствами уже используемого стека.

Не добавлять пакет, если задача решается небольшим количеством понятного кода.

При добавлении зависимости объяснить:

- зачем она нужна;
- почему существующих средств недостаточно;
- какой участок проекта её использует.

Не обновлять все зависимости ради одной задачи.

Не изменять lock-файл без фактического изменения зависимостей.

---

## 13. Модели и глубина рассуждений

Использовать минимально достаточную модель и глубину рассуждений.

Не проводить глубокий архитектурный анализ для:

- изменения текста;
- небольшой правки CSS;
- замены иконки;
- локального исправления обработчика;
- добавления простого поля;
- небольшого исправления конфигурации.

Повышать глубину анализа только для:

- архитектурных изменений;
- сложных ошибок состояния;
- гонок и асинхронных проблем;
- проблем производительности;
- интеграций нескольких подсистем;
- изменений, затрагивающих безопасность или данные.

---

## 14. Итоговый отчёт

Итог должен быть коротким и содержать:

1. Что изменено.
2. Какие файлы изменены.
3. Какие проверки выполнены.
4. Есть ли ограничения или непроверенные моменты.
5. Нужен ли ручной следующий шаг.

Не включать:

- полный журнал команд;
- полные логи;
- пересказ всего процесса;
- повторное описание исходной задачи;
- длинное объяснение очевидных изменений.

Пример:

```text
Готово.

Изменено:
- src/components/VolumeControl.tsx — исправлена обработка энкодера.
- src/hooks/useHelixControl.ts — добавлено ограничение диапазона.

Проверки:
- npm run typecheck — PASS
- npm test -- VolumeControl — PASS

Не выполнял git push.
```

## 15. Порядок действий по умолчанию

Для обычной задачи использовать такой алгоритм:

```text
1. git status --short
2. Найти целевой код через rg
3. Прочитать минимальный контекст
4. Внести точечное изменение
5. Посмотреть diff только изменённых файлов
6. Запустить минимальную релевантную проверку
7. Кратко сообщить результат
```

Не расширять этот процесс без обоснованной необходимости.

---

## 16. Приоритет инструкций

Приоритет правил:

1. Прямой запрос пользователя.
2. Этот файл.
3. Существующие правила и соглашения репозитория.
4. Общие предположения Codex.

Если прямой запрос пользователя конфликтует с безопасностью данных или может уничтожить незакоммиченные изменения, сначала предупредить о риске.

---

## 17. Главный критерий качества

Хорошее выполнение задачи — это не максимальное количество прочитанных файлов, команд или объяснений.

Хорошее выполнение задачи — это:

- минимальный достаточный контекст;
- минимальный безопасный patch;
- релевантная проверка;
- отсутствие побочных изменений;
- краткий и честный результат.
